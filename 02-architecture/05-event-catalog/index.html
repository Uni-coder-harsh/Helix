    HELIX-ARCH-005: Event Catalog

    This document serves as the authoritative, canonical registry for all domain events produced and consumed within the Helix Governance Operating System.

    1. Event Design Standards

    To ensure consistency, readability, and compatibility across all microservices and external plugins, every event registered in this catalog must adhere to the following design standards.

    1.1. Event Naming Rules

    • Tense: Event names must be written in the past tense, representing a historical fact that has already occurred (e.g., IssueValidated, not ValidateIssue).
    • Casing: Event names must use PascalCase format (e.g., CitizenReportSubmitted).
    • Structure: Names must follow the pattern [DomainEntity][ActionPastTense].
    • Good: TaskCompleted, EvidenceAttached.
    • Bad: CompletedTask (inverted), NotifyCitizen (imperative command).
    • Suffixes: Suffixes such as Event or Message are prohibited to avoid redundancy. Suffixes like Command or Query are reserved for synchronous/request payloads and must not be used for asynchronous events.

    1.2. Event Metadata Standard (Standard Envelope)

    Every event emitted in Helix consists of two parts: a standard metadata header envelope (for routing, tracing, and validation) and the event payload.

    The standard metadata envelope contains the following logical attributes:

    Attribute Type Description
    event_id UUIDv4 Universally unique identifier for this specific event instance. Used for deduplication.
    correlation_id UUIDv4 Distributed tracing identifier carried across all downstream events generated by this trigger.
    timestamp ISO-8601 The UTC timestamp when the event was generated by the producer.
    producer_id String Unique identifier of the component or service instance that published the event.
    schema_name String The exact schema identifier matching this event type (e.g., helix.events.workflow.issue_validated).
    schema_version SemVer The semantic version of the schema that this payload complies with (e.g., 1.0.0).
    ordering_key String The partitioning key used to guarantee sequential delivery of related events (e.g., issue_id).
    actor_context Object Conceptual authentication and authorization context of the user or system token that triggered the action. Includes actor_id and roles.

    1.3. Conceptual Event Header Specification

    Below is the conceptual structure of the standard event wrapper:

    Event Wrapper
├── Metadata Header (Envelope)
│   ├── event_id: UUIDv4
│   ├── correlation_id: UUIDv4
│   ├── timestamp: DateTime (UTC)
│   ├── producer_id: String
│   ├── schema_name: String
│   ├── schema_version: String (SemVer)
│   ├── ordering_key: String
│   └── actor_context: Map
│       ├── actor_id: String
│       └── roles: List[String]
└── Payload (Entity-specific domain data)

    3. Event Ownership Matrix

    Event schemas are owned by the Bounded Context responsible for the corresponding entities. Changes to schemas must be approved by the context stewards.

    Bounded Context Schema Owner Service Reviewer / Steward Primary Events Managed
    Citizen Intake WhatsApp/SMS Connectors Intake Lead CitizenReportSubmitted, CitizenOptedOut
    Workflow Engine Workflow Engine Service Core System Architect IssueValidated, EvidenceAttached, DepartmentAssigned, TaskCreated, TaskCompleted, IssueResolved
    Intelligence AI Orchestrator Service AI & Data Lead RecommendationGenerated, EvidenceClustered, AnomalyDetected
    Communications Notification Engine Operations Lead AlertEnqueued, DispatchSucceeded, DispatchFailed
    Identity & Admin Identity & Admin Services Security Lead SchemeConfigured, RoleAssigned, SLAThresholdExceeded
    Audit & Platform Schema Registry / Gateway System Architect SchemaChanged, AccessLogGenerated, SystemConfigOverridden
    Extensibility Plugin Runtime Service Integration Architect HookTriggered, PluginResponseReturned, PluginExecutionFailed

    4. Event Registry

    4.1. Citizen Events

    EVT-101: CitizenReportSubmitted

    • Event Name: CitizenReportSubmitted
    • Category: Citizen Events
    • Producer: WhatsApp Connector, SMS Connector, Citizen Portal
    • Consumers: Workflow Engine, AI Orchestrator
    • Trigger: A citizen successfully sends a text, audio report, or image to Helix via a supported intake channel.
    • Ordering Key: citizen_id
    • PII Classification: Restricted (Contains citizen phone number/account handle and raw submission text).
    • Schema Owner: Citizen Intake Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • citizen_id (Unique identifier or hash of intake handle)
    • channel_type (e.g., WHATSAPP, SMS, WEB)
    • raw_payload_uri (Object storage URI to raw audio, text, or image metadata)
    • submission_timestamp (Raw ingestion date)
    • Description: Emitted immediately when a citizen submission passes edge authentication and validation, indicating new raw content is ready for workflow evaluation.

    EVT-102: CitizenFeedbackSubmitted

    • Event Name: CitizenFeedbackSubmitted
    • Category: Citizen Events
    • Producer: WhatsApp Connector, Citizen Portal
    • Consumers: Analytics Platform, Workflow Engine
    • Trigger: A citizen responds to a completion notification with a rating or text feedback.
    • Ordering Key: issue_id
    • PII Classification: Restricted (Contains reference to user identifier and subjective comments).
    • Schema Owner: Citizen Intake Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • issue_id (The reference resolved issue)
    • rating (Integer 1-5 scale)
    • comments (Optional raw response string)
    • Description: Tracks citizen satisfaction and verification of the provided resolution. Used to calculate department and constituency success metrics.

    EVT-103: CitizenOptedOut

    • Event Name: CitizenOptedOut
    • Category: Citizen Events
    • Producer: Citizen Portal, Administration Portal
    • Consumers: Identity Service, Workflow Engine, Notification Engine
    • Trigger: A citizen requests profile deletion or revokes contact permissions.
    • Ordering Key: citizen_id
    • PII Classification: None (Does not contain PII, but commands its erasure).
    • Schema Owner: Identity & Admin Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • citizen_id (The identifier to be purged)
    • opt_out_timestamp (Date of request)
    • reason_code (Standardized categorical reason for opting out)
    • Description: Triggers the privacy-cleansing (crypto-shredding) sequence across all operational stores.

    4.2. Workflow Events

    EVT-201: IssueValidated

    • Event Name: IssueValidated
    • Category: Workflow Events
    • Producer: Workflow Engine
    • Consumers: AI Orchestrator, Notification Engine, Knowledge Graph Sync
    • Trigger: An incoming report is reviewed and approved as a valid municipal or state issue.
    • Ordering Key: issue_id
    • PII Classification: Private (Contains general location and issue details, citizen identifier).
    • Schema Owner: Workflow Engine Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • issue_id (Newly assigned operational issue ID)
    • citizen_id (Owner citizen identifier)
    • category (Classified category of issue, e.g., WATER_LEAKAGE)
    • location (Geo-coordinates and ward mapping)
    • severity (Calculated priority status)
    • Description: Represents the transition from a raw, unverified submission to a formal operational issue in the Helix registry.

    EVT-202: EvidenceAttached

    • Event Name: EvidenceAttached
    • Category: Workflow Events
    • Producer: Mobile App (Field Agent), Citizen Portal, Workflow Engine
    • Consumers: AI Orchestrator, Knowledge Graph Sync
    • Trigger: A field worker uploads validation photographs or a citizen uploads extra context for an active issue.
    • Ordering Key: issue_id
    • PII Classification: Private (May contain images containing faces, license plates, or location clues).
    • Schema Owner: Workflow Engine Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • issue_id (The reference issue ID)
    • evidence_id (Unique identifier for this artifact)
    • uploader_id (Identifier of actor who attached it)
    • file_uri (Object storage URI of evidence asset)
    • file_type (e.g., IMAGE, PDF, AUDIO)
    • Description: Logs that additional supporting materials have been permanently associated with the active issue, triggering re-evaluation of routing or severity.

    EVT-203: DepartmentAssigned

    • Event Name: DepartmentAssigned
    • Category: Workflow Events
    • Producer: Workflow Engine, AI Orchestrator (with human verification)
    • Consumers: Analytics Platform, Knowledge Graph Sync, Notification Engine
    • Trigger: An issue is assigned to a specific department or municipal unit for resolution.
    • Ordering Key: issue_id
    • PII Classification: Private (Contains reference to assigned officer/department IDs).
    • Schema Owner: Workflow Engine Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • issue_id (The reference issue ID)
    • department_id (The assigned organizational unit ID)
    • assigned_officer_id (Optional specific supervisor ID)
    • sla_duration_seconds (Allotted resolution timeline)
    • Description: Transitions accountability of the issue to a specific administrative authority.

    EVT-204: TaskCreated

    • Event Name: TaskCreated
    • Category: Workflow Events
    • Producer: Workflow Engine
    • Consumers: Notification Engine, Mobile App Gateway
    • Trigger: A concrete work order or physical task is spawned under an issue.
    • Ordering Key: issue_id
    • PII Classification: Private (Contains instructions, locations, and worker details).
    • Schema Owner: Workflow Engine Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • task_id (Unique task identifier)
    • issue_id (Parent issue identifier)
    • assigned_team_id (Field team assigned to perform work)
    • description (Detailed engineering description of work required)
    • scheduled_start (Proposed start date)
    • Description: Represents a physical action item assigned to field crews (e.g. "Replace pipe at ward 4").

    EVT-205: TaskCompleted

    • Event Name: TaskCompleted
    • Category: Workflow Events
    • Producer: Mobile App (Field Agent)
    • Consumers: Workflow Engine, Analytics Platform
    • Trigger: A field worker completes the work order and submits completion photos/records.
    • Ordering Key: issue_id
    • PII Classification: Private (May contain worker signatures, site photos).
    • Schema Owner: Workflow Engine Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • task_id (Reference task identifier)
    • issue_id (Parent issue identifier)
    • completion_timestamp (Completion date)
    • resolution_summary (Notes describing the physical fix)
    • evidence_uris (URIs to validation photos)
    • Description: Signals that the physical remediation work is done. It triggers the Workflow Engine to transition the parent issue status.

    EVT-206: IssueResolved

    • Event Name: IssueResolved
    • Category: Workflow Events
    • Producer: Workflow Engine
    • Consumers: Notification Engine, Analytics Platform, Knowledge Graph Sync
    • Trigger: The administrative supervisor approves the resolution and closes the issue.
    • Ordering Key: issue_id
    • PII Classification: Private (Contains details of final resolution, citizen contact references).
    • Schema Owner: Workflow Engine Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • issue_id (The closed issue ID)
    • resolution_timestamp (Closure date)
    • resolution_type (e.g., FIXED, DUPLICATE, OUT_OF_JURISDICTION)
    • final_report_uri (URI to the comprehensive summary document)
    • Description: The terminal state of a primary Helix workflow, indicating resolution approval and triggering feedback loops.

    4.3. AI Events

    EVT-301: RecommendationGenerated

    • Event Name: RecommendationGenerated
    • Category: AI Events
    • Producer: AI Orchestrator Service
    • Consumers: Workflow Engine, Admin Portal (UI suggestion panel)
    • Trigger: An AI model processes raw issue text/photos and suggests routes or priorities.
    • Ordering Key: issue_id
    • PII Classification: Private (Contains recommendations tied to specific locations and citizen reports).
    • Schema Owner: Intelligence Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • issue_id (The reference issue ID)
    • recommendation_type (e.g., ROUTING, PRIORITY, PREPARED_REPLY)
    • suggested_values (Map containing proposed options like department IDs, SLA durations)
    • confidence_score (Float value representing model confidence, 0.0 - 1.0)
    • grounding_references (List of knowledge graph node identifiers used for grounding)
    • Description: Broadcasts an advisory suggestion. These are caught by the Workflow Engine and queued for human operator verification.

    EVT-302: EvidenceClustered

    • Event Name: EvidenceClustered
    • Category: AI Events
    • Producer: AI Orchestrator Service (Analytics clustering module)
    • Consumers: Workflow Engine, Analytics Platform, Knowledge Graph Sync
    • Trigger: Machine learning models group multiple individual citizen complaints into a single root-cause cluster.
    • Ordering Key: cluster_id
    • PII Classification: None (Abstract cluster mappings, references only issue IDs).
    • Schema Owner: Intelligence Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • cluster_id (Unique identifier for the structural problem)
    • associated_issue_ids (List of grouped citizen issue IDs)
    • centroid_location (Average geographical coordinate)
    • confidence_level (Float value representing cluster density)
    • Description: Groups duplicate complaints together, facilitating a single municipal fix rather than scattered dispatch actions.

    EVT-303: AnomalyDetected

    • Event Name: AnomalyDetected
    • Category: AI Events
    • Producer: AI Orchestrator Service (Guardrail checks)
    • Consumers: Workflow Engine, Security Audit Log
    • Trigger: Ingestion guardrails detect malicious activity, policy conflicts, or fraud patterns.
    • Ordering Key: issue_id
    • PII Classification: Restricted (Contains flagged raw payloads).
    • Schema Owner: Intelligence Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • issue_id (Reference issue ID)
    • anomaly_type (e.g., FRAUD_PROJECTION, PROMPT_INJECTION, DUPLICATE_SPAM)
    • flagged_details (Reasoning behind the anomaly classification)
    • Description: Flags suspicious reports immediately for quarantine, isolating the system core from spam or security injection vectors.

    4.4. Notification Events

    EVT-401: AlertEnqueued

    • Event Name: AlertEnqueued
    • Category: Notification Events
    • Producer: Workflow Engine, Administration Service
    • Consumers: Notification Engine
    • Trigger: A workflow change requires an outbound update to a citizen or staff member.
    • Ordering Key: recipient_id
    • PII Classification: Restricted (Contains recipient contact handles, names, custom details).
    • Schema Owner: Communications Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • notification_id (Unique identifier for the message queue)
    • recipient_id (Target citizen or staff ID)
    • channel (e.g., WHATSAPP, SMS, EMAIL)
    • template_name (Target messaging structure to apply)
    • template_variables (Map of variables to hydrate text)
    • Description: Represents a request to dispatch a user-facing update. Captured by the Notification Engine.

    EVT-402: DispatchSucceeded

    • Event Name: DispatchSucceeded
    • Category: Notification Events
    • Producer: Notification Engine (Connector gateway)
    • Consumers: Workflow Engine, Analytics Platform
    • Trigger: An external delivery gateway (e.g., Twilio, SMTP) confirms message receipt.
    • Ordering Key: recipient_id
    • PII Classification: Private (References recipient IDs and delivery receipts).
    • Schema Owner: Communications Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • notification_id (Reference message identifier)
    • gateway_receipt (Receipt handle from delivery channel)
    • delivered_timestamp (Confirmation date)
    • Description: Closes the feedback loop for communications, proving the citizen was successfully updated.

    EVT-403: DispatchFailed

    • Event Name: DispatchFailed
    • Category: Notification Events
    • Producer: Notification Engine (Connector gateway)
    • Consumers: Workflow Engine, Operational Alerting
    • Trigger: Outbound delivery fails due to invalid credentials, rate limit, or phone disconnection.
    • Ordering Key: recipient_id
    • PII Classification: Private (References recipient IDs, gateway error logs).
    • Schema Owner: Communications Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • notification_id (Reference message identifier)
    • error_code (Gateway-specific error code)
    • error_message (Detailed description of delivery failure)
    • retry_count (Attempts made before failure confirmation)
    • Description: Flags communication breakdowns. Allows workflow handlers to route updates to alternative channels.

    4.5. Administration Events

    EVT-501: SchemeConfigured

    • Event Name: SchemeConfigured
    • Category: Administration Events
    • Producer: Administration Service (Operator portal)
    • Consumers: Workflow Engine, AI Orchestrator, Knowledge Graph Sync
    • Trigger: An administrator creates or changes a government scheme policy or validation rule.
    • Ordering Key: scheme_id
    • PII Classification: None (Public policy rules).
    • Schema Owner: Identity & Admin Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • scheme_id (Unique scheme identifier)
    • scheme_name (Public title of scheme)
    • rules_definition_uri (URI mapping to operational routing rules and JSON policy structures)
    • is_active (Boolean active state flag)
    • Description: Broadcasts configuration changes. Consuming services update their caches to enforce new policy parameters.

    EVT-502: RoleAssigned

    • Event Name: RoleAssigned
    • Category: Administration Events
    • Producer: Identity Service (Operator portal)
    • Consumers: API Gateway, All Services (RBAC caches)
    • Trigger: Permissions of an administrative user or officer are modified.
    • Ordering Key: user_id
    • PII Classification: Private (References user IDs and operational security clearances).
    • Schema Owner: Identity & Admin Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • user_id (Target user identifier)
    • role_names (List of newly assigned active security roles)
    • jurisdiction_boundaries (Map representing assigned ward or district limits)
    • Description: Real-time synchronization of security bounds across distributed nodes.

    EVT-503: SLAThresholdExceeded

    • Event Name: SLAThresholdExceeded
    • Category: Administration Events
    • Producer: Workflow Engine (Cron/Tick service)
    • Consumers: Analytics Platform, Operational Alerting, Notification Engine
    • Trigger: An active issue fails to transition to resolved within the legal time allocation.
    • Ordering Key: issue_id
    • PII Classification: Private (Contains reference to stalled issue details and assigned department).
    • Schema Owner: Identity & Admin Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • issue_id (The violating issue ID)
    • assigned_department_id (Owner department of stalled issue)
    • days_overdue (Floating point scale of SLA breach)
    • Description: Critical operational warning. Escalates issues to supervisors and triggers systemic alerts.

    4.6. Audit Events

    EVT-601: SchemaChanged

    • Event Name: SchemaChanged
    • Category: Audit Events
    • Producer: Schema Registry
    • Consumers: All active producers and consumers
    • Trigger: A new version of an event contract is officially registered and approved.
    • Ordering Key: schema_name
    • PII Classification: None (Schema layout strings).
    • Schema Owner: Audit & Platform Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • schema_name (The modified schema identifier)
    • old_version (Previous active version tag)
    • new_version (New version tag)
    • compatibility_mode (e.g. BACKWARD, FORWARD, NONE)
    • Description: Informs microservices that event parsing parameters have changed, triggering schema dynamic cache invalidations.

    EVT-602: AccessLogGenerated

    • Event Name: AccessLogGenerated
    • Category: Audit Events
    • Producer: API Gateway, Core Services
    • Consumers: Security Audit Log Archiver
    • Trigger: A user queries or exports citizen PII records.
    • Ordering Key: actor_id
    • PII Classification: Private (References accessing actor ID and targets accessed).
    • Schema Owner: Audit & Platform Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • log_id (Unique audit log ID)
    • actor_id (The actor requesting information)
    • queried_resource_type (e.g. CITIZEN_PROFILE, PII_EXPORT)
    • resource_id (Reference target entity queried)
    • ip_address (Access IP mapping)
    • Description: Cryptographically signed security audit trail, ensuring zero-trust trace tracking.

    EVT-603: SystemConfigOverridden

    • Event Name: SystemConfigOverridden
    • Category: Audit Events
    • Producer: Administration Service
    • Consumers: All Core Services
    • Trigger: A developer or system operator overrides a critical system runtime constant or environment configuration.
    • Ordering Key: config_key
    • PII Classification: Private (Mentions config parameters and actor details).
    • Schema Owner: Audit & Platform Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • config_key (The modified key)
    • previous_value_hash (Secure hash of previous config state)
    • new_value_hash (Secure hash of new config state)
    • actor_id (The administrator who applied the override)
    • Description: Logs direct operational mutations, critical for tracing systemic regressions.

    4.7. Plugin Events

    EVT-701: HookTriggered

    • Event Name: HookTriggered
    • Category: Plugin Events
    • Producer: Workflow Engine, API Gateway
    • Consumers: Plugin Runtime Service
    • Trigger: Core workflows reach a registered hook extension boundary (e.g., custom geo-routing calculations).
    • Ordering Key: issue_id
    • PII Classification: Private (Carries raw contextual arguments of workflow entities).
    • Schema Owner: Extensibility Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • hook_execution_id (Unique lifecycle execution identifier)
    • plugin_id (Target extension plugin configured)
    • hook_name (The target function hook, e.g. pre_route_validation)
    • context_payload (State Map injected into the plugin sandbox)
    • Description: Commands the Plugin Runtime to initialize sandboxed subprocesses or WASM instances for custom logic.

    EVT-702: PluginResponseReturned

    • Event Name: PluginResponseReturned
    • Category: Plugin Events
    • Producer: Plugin Runtime Service
    • Consumers: Workflow Engine, API Gateway
    • Trigger: An isolated plugin completes its task successfully and reports results back.
    • Ordering Key: issue_id
    • PII Classification: Private (Contains outputs of custom calculations on citizen data).
    • Schema Owner: Extensibility Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • hook_execution_id (The reference execution trigger ID)
    • plugin_id (Executing plugin identifier)
    • execution_duration_ms (Time spent in sandbox)
    • response_payload (Output data returned by plugin)
    • Description: Notifies core engines that third-party calculations are finished, injecting results back to the workflow.

    EVT-703: PluginExecutionFailed

    • Event Name: PluginExecutionFailed
    • Category: Plugin Events
    • Producer: Plugin Runtime Service
    • Consumers: Workflow Engine, Operational Alerting
    • Trigger: A sandboxed plugin throws an error, exceeds CPU/Memory limits, or times out.
    • Ordering Key: issue_id
    • PII Classification: None (Mentions execution diagnostics only).
    • Schema Owner: Extensibility Bounded Context
    • Version: 1.0.0
    • Payload Summary:
    • hook_execution_id (The reference execution trigger ID)
    • plugin_id (Failing plugin identifier)
    • error_class (e.g., TIMEOUT, OUT_OF_MEMORY, UNCAUGHT_EXCEPTION)
    • diagnostic_message (Sandbox error logs)
    • Description: Signals plugin failures to the core coordinator to trigger fallback default pathways (such as manual routing).

    5. Event Governance Policies

    5.1. Version Compatibility Rules

    Every schema change registered in the Helix registry must follow these strict compatibility parameters:

    1. Backward Compatibility (Default REQUIREMENT): Consuming applications running older code must be able to process new events without crashing. Adding optional fields or metadata parameters are fully backward-compatible changes.
    2. Forward Compatibility: Older events must be readable by updated consumer code. Removing optional fields or adding default values to required fields ensures forward compatibility.
    3. Breaking Changes: If a schema must undergo breaking changes (e.g., changing data type formats, making optional fields required, or removing fields):
    4. A new major schema version must be registered (e.g., 1.0.0 ➔ 2.0.0).
    5. The event name remains unchanged, but the Event Metadata version updates.
    6. For a transition period, the event bus must support automated adapters that translate 2.0.0 payloads to 1.0.0 for legacy consumers.

    5.2. Deprecation Policy

    When an event schema is replaced by a newer version or retired, it enters the Deprecation Lifecycle:

    [Active Schema] ────► [Deprecated] ────► [Grace Period (180 days)] ────► [Sunset / Retired]
    1. Notification: The Schema owner triggers a SchemaChanged event detailing the deprecation plan.
    2. Grace Period: The schema remains fully supported for a grace period of 180 days or two minor system versions, whichever is longer. Consuming teams must migrate to the new schema during this window.
    3. Sunset: After the grace period, the Schema Registry rejects validation queries for the deprecated version, and producers cease dual-publication.

    6. Event Schema Validation Checklist

    Before submitting an addition or change to this catalog to the Architecture Review Board, verify the following details:

    • Past Tense Name: Is the event named using PascalCase past-tense format (DomainEntityAction)?
    • Ordering Key Defined: Is there an explicit partition ordering key designated to prevent race conditions during concurrent consumer runs?
    • PII Mapped: Are all fields containing sensitive personal details declared as Private/Restricted, and is the key encryption mechanism defined?
    • Schema Owner Assigned: Is a specific microservice context and team steward assigned to this schema definition?
    • Traceability: Does the payload carry standard identifier references to correlate it with the parent issue_id or citizen_id?
    • Semantic Versioning: Is the starting schema version marked as 1.0.0 for new events, or is the increment correctly typed (Major/Minor/Patch) for modifications?
